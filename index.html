It's a case of semantics

Why use semantic elements?

Some of the benifits from writing semantic markup are as follows:

  • Search engines will consider its contents as important keywords to influence the page's search rankings (see SEO)
  • Finding blocks of meaningful code is significantly easier than searching through endless divs with or without semantic or namespaced classes
  • Suggests to the developer the type of data that will be populated
  • Semantic naming mirrors proper custom element/component naming

Source: MDN on Semantics

What elements should I use?

When approaching which markup to use, ask yourself, "What element(s) best describe/represent the data that I'm going to populate?" For example, is it a list of data?; ordered, unordered?; is it an article with sections and an aside of related information?; does it list out definitions?; is it a figure or image that needs a caption?; should it have a header and a footer in addition to the global site-wide header and footer?; etc.

Source: MDN on Semantics

Some generic HTML elements

Block level element

div:The HTML Content Division is the generic container for flow content. It has no effect on the content or layout until styled using CSS.

Inline level element

span:The HTML element is a generic inline container for phrasing content, which does not inherently represent anything. It can be used to group elements for styling purposes(using the class or id attributes), or becuase the share attribute values, such as lang.

Source: MDN on Elements
